🔐 Personal app deployment guide

Run a FastAPI or Node.js app on a DigitalOcean Droplet and access it securely from anywhere.

A practical, easy-to-follow guide for personal projects: private access with Tailscale Serve, public access with Caddy, and a sane firewall posture so your server does not become an internet piñata.

Browser on laptop / phone
↓
Tailscale Serve or Caddy HTTPS endpoint
↓
Your app on 127.0.0.1:8000 or 127.0.0.1:3000
Best rule of thumb

Private tools use Tailscale. Public apps use Caddy.

If only you should access it, keep it on your tailnet. If visitors, recruiters, or random browser goblins should access it, put Caddy in front of it with a real domain.

1. Choose the right access model

Decision table
Goal Recommended setup Why
Personal dashboard, admin panel, prototype, private API Tailscale Serve Accessible only to your tailnet devices. No public ports required.
Portfolio project, public demo, website, recruiter-facing app Caddy reverse proxy Clean public HTTPS with your own domain or subdomain.
Temporary public demo without full DNS setup Tailscale Funnel Can expose a local service publicly, but use carefully and understand limits.

2. Prepare the Droplet

Base server hygiene
  • Use SSH keys instead of password login where possible.
  • Run updates regularly.
  • Use a firewall: DigitalOcean Cloud Firewall, UFW, or both.
  • Expose only what you need.
Typical ports
22SSH access
80/443Public web traffic if using Caddy
3000/8000App ports. Keep local only.

Update packages

sudo apt update
sudo apt upgrade

Optional UFW baseline

For a private Tailscale-only app, you generally do not need to open your app port publicly. For a public Caddy app, allow HTTP and HTTPS.

sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
sudo ufw status verbose

Do not blindly enable UFW if you are unsure SSH is allowed. Locking yourself out is the server-admin version of stepping on a rake.

3. Run your app locally on the Droplet

Your app must already be running before Tailscale Serve or Caddy can proxy to it. The proxy is the doorway, not the person living in the apartment.

FastAPI example

Run FastAPI with Uvicorn on localhost:

cd ~/my-fastapi-app
source .venv/bin/activate
uvicorn main:app --host 127.0.0.1 --port 8000

Test locally:

curl http://127.0.0.1:8000
Node.js example

Run a Node app on localhost:

cd ~/my-node-app
npm install
HOST=127.0.0.1 PORT=3000 npm start

Test locally:

curl http://127.0.0.1:3000

Bind to 127.0.0.1 for proxy setups. That means the app listens only on the Droplet itself. Tailscale Serve or Caddy then handles access from outside.

4. Private access with Tailscale Serve

Use this when the app is only for you and your tailnet devices. Tailscale Serve routes traffic from your tailnet to a local service running on the Droplet.

Confirm the Droplet is on your tailnet

tailscale status

You should see the Droplet listed with a 100.x.y.z Tailscale IP.

Find your Tailscale hostname

tailscale status --self

Example hostname:

jimjamscozz-droplet-1.tail837b53.ts.net

Serve a FastAPI app on port 8000

tailscale serve --bg http://127.0.0.1:8000

Then open:

https://jimjamscozz-droplet-1.tail837b53.ts.net

Or serve a Node app on port 3000

tailscale serve --bg http://127.0.0.1:3000

Check Serve status

tailscale serve status

This shows which local app/path Tailscale is currently forwarding to.

Multiple private apps on one Droplet

tailscale serve --bg /api http://127.0.0.1:8000
tailscale serve --bg /app http://127.0.0.1:3000

Then use:

https://your-droplet.tailnet.ts.net/api
https://your-droplet.tailnet.ts.net/app

Tailscale Serve does not start your app. Your FastAPI or Node process must be running on the target port, or Serve will proxy into the void.

5. Public access with Caddy

Use this when you want a public website or API with a normal domain such as app.example.com. Caddy is excellent because it handles reverse proxying and automatic HTTPS when it knows the hostname.

Create a DNS record

Point a subdomain to your Droplet’s public IPv4 address:

app.example.com  A  143.198.119.132

Install Caddy

Use the official Caddy package instructions for Ubuntu/Debian. After installation, Caddy usually runs as a system service.

Edit the Caddyfile

sudo nano /etc/caddy/Caddyfile

FastAPI example:

app.example.com {
    reverse_proxy 127.0.0.1:8000
}

Node example:

app.example.com {
    reverse_proxy 127.0.0.1:3000
}

Reload Caddy

sudo caddy validate --config /etc/caddy/Caddyfile
sudo systemctl reload caddy

Public app pattern: Caddy listens on 80/443, your app listens on localhost, and visitors never touch your raw app port directly.

6. Make the app auto-start with systemd

For anything you care about, do not rely on a manually opened terminal. Make Linux babysit the process.

FastAPI systemd service 
sudo nano /etc/systemd/system/my-fastapi-app.service
[Unit]
Description=My FastAPI App
After=network.target

[Service]
User=YOUR_USER
WorkingDirectory=/home/YOUR_USER/my-fastapi-app
Environment="PATH=/home/YOUR_USER/my-fastapi-app/.venv/bin"
ExecStart=/home/YOUR_USER/my-fastapi-app/.venv/bin/uvicorn main:app --host 127.0.0.1 --port 8000
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
Node systemd service 
sudo nano /etc/systemd/system/my-node-app.service
[Unit]
Description=My Node App
After=network.target

[Service]
User=YOUR_USER
WorkingDirectory=/home/YOUR_USER/my-node-app
Environment=HOST=127.0.0.1
Environment=PORT=3000
ExecStart=/usr/bin/npm start
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target

Enable and start the service

sudo systemctl daemon-reload
sudo systemctl enable --now my-fastapi-app
sudo systemctl status my-fastapi-app

For the Node example, replace my-fastapi-app with my-node-app.

View logs

journalctl -u my-fastapi-app -f

7. Security checklist

  • Keep the app bound to 127.0.0.1 unless you intentionally need public binding.
  • Do not open app ports like 3000 or 8000 to the public internet.
  • Use Tailscale Serve for private tools and Caddy for public apps.
  • For public Caddy apps, open only 80/tcp and 443/tcp for web traffic.
  • Use environment variables or secret files for API keys. Never hardcode secrets into GitHub repos.
  • Add authentication inside the app if it exposes private data, even if it is tailnet-only.
  • Use systemctl status, journalctl, and tailscale serve status when debugging.
  • Back up important app data, especially SQLite databases and uploaded files.

8. Troubleshooting map

Problem Check Useful command
Browser shows connection error Is the app running locally? curl http://127.0.0.1:8000
Tailscale URL does not work Is Serve pointing to the right port? tailscale serve status
Caddy domain does not work DNS and Caddy config sudo caddy validate --config /etc/caddy/Caddyfile
App dies after reboot Is systemd enabled? systemctl is-enabled my-app
Port confusion goblin appears What is listening? ss -tulpn

Recommended personal-project setup

Private default

Tailscale Serve

Best for dashboards, private utilities, internal APIs, and experiments you do not want exposed to the public internet.

Public default

Caddy

Best for portfolio projects, public demos, and web apps with a custom domain and automatic HTTPS.

Reliability layer

systemd

Best for keeping the app running after SSH disconnects, crashes, or server reboots.
